# Live TrueNAS runtime can fail without truenas_api_client installed

## Summary

Live NAS Reconcile can fail with `ModuleNotFoundError: No module named 'truenas_api_client'` when the dedicated fortress Python runtime is absent or incomplete and the workflow falls back to the ambient Python interpreter.

## Reproduction

From the repo root in an environment without `truenas_api_client` installed in the active Python:

```bash
FORTRESS_NAS_RECONCILE_TRUENAS_TOKEN=dummy-token \
  python3 -m fortress_nas.truenas_reality \
  truenas 127.0.0.1 FORTRESS_NAS_RECONCILE_TRUENAS_TOKEN
```

Observed:

```text
failed to query TrueNAS reality for NAS Endpoint truenas: No module named 'truenas_api_client'
```

The same failure can surface through `scripts/nas-reconcile-plan --live truenas` after SOPS credential extraction if `/opt/fortress-python/bin/python3` does not exist or does not have the TrueNAS API client installed.

## Agent Brief

**Category:** bug
**State:** ready-for-agent
**Summary:** Make live NAS Reconcile fail clearly, or self-check early, when the official TrueNAS API client runtime dependency is missing.

**Current behavior:**
`scripts/nas-reconcile-plan` invokes the live reality loader with `_live_reality_python()`. That helper prefers `FORTRESS_PYTHON`, then `/opt/fortress-python/bin/python3`, and otherwise falls back to `sys.executable`.

`fortress_nas.truenas_client.LiveTrueNasClient.connect()` imports `truenas_api_client.Client` lazily only when a real live client is needed. If the selected interpreter lacks the dependency, the child module reports a generic live query failure ending in `No module named 'truenas_api_client'`.

The project docs already say the official `truenas_api_client` package belongs in the dedicated `/opt/fortress-python` venv and that live NAS Reconcile should use that interpreter when present.

**Desired behavior:**
The operator should get a clear, actionable error before live reconciliation logic proceeds when the TrueNAS API client runtime is unavailable.

The fix should preserve the existing dependency model:

- Fixture-backed `--reality-json` mode must not require `truenas_api_client`.
- Unit tests that inject fake TrueNAS clients should not require the real package.
- Live mode should continue to prefer `FORTRESS_PYTHON`, then `/opt/fortress-python/bin/python3`.
- If the selected live runtime cannot import `truenas_api_client`, the command should name the missing runtime dependency and point to the fortress toolchain/runtime setup rather than presenting it as an ordinary TrueNAS query failure.

**Codebase context:**
- `scripts/nas-reconcile-plan` chooses the child interpreter in `_live_reality_python()` and launches `python -m fortress_nas.truenas_reality`.
- `fortress_nas/truenas_client.py` imports `truenas_api_client.Client` lazily in `LiveTrueNasClient.connect()`.
- `fortress_nas/truenas_reality.py` catches generic exceptions from `load_live_truenas_reality()` and currently prints `failed to query TrueNAS reality for NAS Endpoint ...`.
- `scripts/setup/install-toolchain.sh` creates `/opt/fortress-python` and installs `git+https://github.com/truenas/api_client.git@${TRUENAS_API_CLIENT_TAG}`.
- `docs/toolchain.md` documents `truenas_api_client` as a runtime library installed into `/opt/fortress-python`.

**Acceptance criteria:**
- [ ] Running live TrueNAS reality loading without `truenas_api_client` installed produces an actionable error that names the missing dependency and the expected fortress Python runtime.
- [ ] `scripts/nas-reconcile-plan --live <endpoint>` preserves credential redaction while surfacing that actionable dependency error from the child process.

**Suggested implementation notes:**
