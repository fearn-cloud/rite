# Take Forgejo backup gate

## What to do

Take and verify a consistent backup immediately before the Forgejo Upgrade maintenance work proceeds. This issue is human-owned because it requires control over the live VM/storage backup path and a judgment call between VM snapshot and minimum application-level dump.

The preferred backup is a stopped-Forgejo VM snapshot or equivalent `forgejo-vm` disk backup. If that is unavailable, use the runbook's application-level `forgejo dump` fallback during the same maintenance window.

## Acceptance criteria

- [x] The chosen backup method is recorded: preferred VM snapshot/disk backup or minimum application-level `forgejo dump`.
- [x] Forgejo is stopped before the consistent backup is taken, unless the chosen platform snapshot mechanism explicitly guarantees consistency another way and that reasoning is recorded.
- [x] The backup artifact identity is recorded, such as snapshot name, backup job id, path, timestamp, or dump filename.
- [x] The backup covers the Service-owned Forgejo data mounted into the container at `/data`.
- [x] The operator has confirmed how to restore the backup before proceeding with the upgrade.
- [x] If the maintenance is paused after the backup, Forgejo is deliberately restarted and that decision is recorded.
