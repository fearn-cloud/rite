# Forgejo runners use a dedicated VM, native runner, and Podman

Forgejo Actions runs trusted operator-authored CI code, but CI jobs are still powerful runtime activity, so runner workloads run on a dedicated Forgejo Runner VM rather than on the Forgejo VM or a Proxmox Host. The runner is installed as a Native systemd service using the dedicated runner user's Podman socket for job containers; the runner itself is not deployed as a Quadlet Service because runner-as-container would require socket mounting, privileged nested containers, or Docker-in-Docker-style plumbing to create sibling job containers. Phase-one Forgejo Actions are limited to non-mutating repository validation: job containers do not receive container-runtime socket access, the Runner VM is not an age Recipient and does not decrypt Fortress SOPS secrets, and the first runner allows one concurrent job so resource pressure remains legible. Live Host, VM, NAS, PBS, and Service convergence remains Operator-owned; phase-one CI labels describe validation capabilities only and must not imply deployment or management authority. The runner can reach Forgejo, DNS/NTP, and public dependency sources, but it is not a general Trusted client and does not receive direct management reachability to Proxmox Hosts, NAS, PBS, or ordinary Service Backend ports. The runner VM is backup-excluded and stateless by contract: registration, configuration, labels, and cleanup policy must be reproducible from declared state, while workspaces, image caches, and job artifacts are disposable. Deployment runners require a separate later design.
