# Service Groups are logical; Service Networks own Podman networking

Fortress separates logical operator grouping from VM-local runtime networking: `service_group` names a Service Group that can be launched as a coherent operator target, while `service_network` names the shared Podman network and Container Alias namespace used for private Service-to-Service communication. This avoids making group launch semantics accidentally grant network reachability, preserves Service Groups as ordinary domain language, and keeps the Podman-specific concept explicit in Inventory and validation.

Service Group Launch targets a logical Service Group through `service-group-launch <group>`, requires the launched Services to share one Backend VM for the first implementation, reads that VM's explicit launch order, runs VM Lifecycle Convergence once, deploys every group Service in order, and performs Ingress Regeneration once at the end when needed. The rejected alternatives were continuing to overload `service_group` as the networking field, which would conflict with the operator meaning of a group, and creating standalone Service Group entities before groups have identity beyond a Backend VM's launch choreography.
