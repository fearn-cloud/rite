## What to build

Build a first-class Service-layer Acceptance Test workflow that generates two temporary Operational VMs, provisions them through the ordinary VM lifecycle, reconciles an Ephemeral Dataset and Derived NFS Share, generates a temporary multi-container Quadlet Service with a Service Secret, deploys it through ordinary `service-deploy`, and verifies the Service from the Peer Acceptance VM.

The Primary Acceptance VM hosts the generated Service. The Peer Acceptance VM verifies the Service from outside the Backend VM by curling the Service's direct Backend Published Port on the Primary Acceptance VM. The Service should use a Share-backed Volume from the acceptance NFS Mount and serve a pre-written marker file from that Mount so the peer response proves Service deployment, direct exposure, and mounted content consumption in one path.

The operator-facing command should be shaped like:

```sh
just acceptance-service-layer host=<host> template=<template> endpoint=<nas-endpoint> auto_confirm=false keep_on_fail=false
```

It should delegate to:

```sh
scripts/acceptance-service-layer host=<host> template=<template> endpoint=<nas-endpoint> auto_confirm=<true|false> keep_on_fail=<true|false>
```

## Notes

- Use the glossary terms **Primary Acceptance VM** and **Peer Acceptance VM**; avoid "secondary VM" in operator-facing prose.
- This workflow owns a dedicated Acceptance Policy and dedicated Ephemeral Dataset; it should not reuse `nfs-shared-mount.yaml` or `acceptance-nfs-demo`.
- Direct peer reachability is proven through the generated Service's Backend port, not through a separate test-only Published Port.
- `bind: 0.0.0.0` is intentional here because the Acceptance Test is proving explicit direct VM exposure.
- Service Secrets are in scope because this Acceptance Test is meant to prove the end-to-end Service deployment ceremony.
- Secret verification must avoid leaking plaintext into stdout, stderr, command logs, or test fixtures.
- Do not create an ADR for this workflow unless implementation uncovers a new hard-to-reverse trade-off. The current design follows existing Acceptance Test, Ephemeral Dataset, Derived Share, VM lifecycle, Quadlet, and Service Secret decisions.
