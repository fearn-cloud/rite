# Service Runtime Intent: Backend and Published Port analysis

Status: ready-for-human
Category: enhancement
Date: 2026-05-16

## Agent Brief

**Category:** enhancement
**Summary:** Add the first Service Runtime Intent tracer bullet for Backend and Published Port analysis.

**Current behavior:**
Service runtime meaning is split across Inventory validation, Service Deploy helpers, and Quadlet rendering helpers. Callers still traverse raw Service yaml to answer fortress-owned runtime questions such as which Backend a Service uses, which Published Ports exist on a Backend VM, whether a Backend port is matched by an Ingress-capable Published Port, and whether ports collide across Services.

This makes the current modules shallow: validation, deploy preparation, and rendering each need to know nested Service yaml shape and repeat pieces of the same runtime rules.

**Desired behavior:**
Introduce a Service Runtime Intent module that can analyze the fleet and expose Backend and Published Port runtime facts plus diagnostics.

The fleet analysis should be able to return partial facts alongside all diagnostics so Inventory validation can report every relevant error at once. A stricter per-Service query may be added if useful, but deploy behavior does not need to move in this tracer bullet.

Inventory validation should adapt Service Runtime Intent diagnostics into ordinary validation errors. Existing validation behavior and error codes should remain stable unless the existing tests clearly require a more precise diagnostic shape.

**Key interfaces:**
- `Service Runtime Intent` — a read-only projection of fortress-owned Service runtime meaning from Inventory. For this issue, it should cover Backend facts and Published Port facts.
- Fleet analysis function — accepts the loaded Inventory model and returns runtime facts plus diagnostics without running deploy ceremony or rendering Quadlet as its primary output.
- Runtime diagnostics — structured enough for Inventory validation to convert into validation errors with paths, codes, and messages.
- Backend runtime fact — identifies the Service, Backend VM, and Backend port for a singular Backend.
- Published Port runtime fact — identifies the Service, Backend VM, container name or index, host port, container port, bind address, protocol parts, and whether it is marked for Ingress.

**Acceptance criteria:**
- [x] The codebase has a Service Runtime Intent module documented or named with the `Service Runtime Intent` domain term from `CONTEXT.md`.
- [x] Fleet analysis resolves Backend runtime facts for Services with singular Backends.
- [x] Fleet analysis normalizes Quadlet Published Ports, including default host port, default bind address, default protocol, and `tcp_udp` expansion.
- [x] Fleet analysis diagnoses Backend port collisions between Services on the same Backend VM.
- [x] Fleet analysis diagnoses Published Port collisions between Services on the same Backend VM for each protocol.
- [x] Fleet analysis diagnoses Ingress-enabled Quadlet Services whose Backend port is not matched by exactly one TCP-capable Published Port marked for Ingress.
- [x] Inventory validation uses Service Runtime Intent diagnostics for the Backend and Published Port rules covered by this issue.
- [x] Existing operator-facing behavior for Service Deploy, Quadlet rendering, and Ingress Regeneration remains unchanged.
- [x] Tests cover the new Service Runtime Intent seam directly and preserve existing Inventory validation coverage for the migrated rules.

**Out of scope:**
- Moving Service Deploy vars to Service Runtime Intent.
- Moving Service Data Directories, Service Secrets, Share-backed Volumes, Service Group runtime wiring, or container start order.
- Changing Quadlet rendering output.
- Running deploy ceremony, decrypting SOPS material, invoking Ansible, or changing operator workflow scripts.
- Replacing flat per-Entity Inventory yaml as the source of truth.
