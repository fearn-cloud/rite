# Live TrueNAS client preflight

## What to build

Introduce a thin fortress adapter around the official `truenas_api_client` package for live NAS Reconcile and run a non-mutating credential preflight before building a live plan. Fortress should not implement the WebSocket/JSON-RPC protocol itself; the adapter owns only fortress-specific method calls, error translation, and test seams.

## Acceptance criteria

- [x] Live mode connects through `truenas_api_client.Client` to the selected NAS Endpoint `management_address` using the token value exported by issue 01.
- [x] The TrueNAS API client dependency is installed/pinned through the project toolchain, preferably using a TrueNAS-version-matched stable tag.
- [x] Preflight verifies the management API is reachable and Dataset/NFS Share read capabilities are available before reconciliation logic runs.
- [x] Apply mode checks NFS Share write capability only if TrueNAS exposes a safe non-mutating permission check.
- [x] Preflight never creates, updates, or deletes a Dataset or Share as a credential test.
- [x] Connection failure, invalid credential, and insufficient privilege failures name the failed capability class without printing the credential value.
- [x] Tests fake the fortress adapter boundary and do not require a live TrueNAS system.

## Agent Brief

**Category:** enhancement
**Summary:** Add the live TrueNAS client adapter and non-mutating preflight required before live NAS Reconcile can build a plan.

**Current behavior:**
Live NAS Reconcile has an endpoint-explicit command surface and can prepare the NAS Reconcile Credential from the selected NAS Endpoint's Sibling SOPS File into the endpoint-declared environment variable. The live reality loading path is still too direct: fortress does not yet have a durable adapter boundary around the official TrueNAS API client, the dependency is not pinned through the project toolchain, and live mode does not perform a clear non-mutating capability preflight before reconciliation logic runs.

**Desired behavior:**
Live NAS Reconcile should connect to the selected NAS Endpoint's Management Address using the official `truenas_api_client.Client` package and the NAS Reconcile Credential already exported by the live credential entrypoint. Before building a live NAS Reconcile Plan, fortress should run a non-mutating preflight that proves the management API is reachable and that the credential can read Dataset metadata and NFS Share definitions.

Failures should be operator-readable and should distinguish connection failure, invalid credential, and insufficient capability by capability class. No error, plan output, log, or test helper output should print the NAS Reconcile Credential value. Apply mode should check NFS Share write capability only if TrueNAS exposes a safe non-mutating check; if no such check exists, this issue should document that no write preflight was added rather than testing write access by mutating a Share.

**Key interfaces:**
- Live NAS Reconcile command surface — `--live <endpoint>` should perform preflight after credential preparation and before live reality is mapped into a NAS Reconcile Plan.
- NAS Endpoint Inventory shape — the endpoint's `management_address` is the API target and `api_token_env` names the process-local credential source.
- TrueNAS client adapter — a thin fortress-owned boundary around `truenas_api_client.Client` that owns fortress-specific method calls, error translation, and test seams; fortress should not implement the WebSocket/JSON-RPC protocol itself.
- TrueNAS API client dependency — installed and pinned through the project toolchain, preferably to a stable tag matched to the operator's intended TrueNAS SCALE API generation.
- Preflight result/errors — should name failed capability classes such as management API reachability, Dataset read, NFS Share read, and optional NFS Share write preflight without leaking credential material.
