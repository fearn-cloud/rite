## What to build

Implement first-class Service Launch as the operator workflow for making one declared Service usable from Inventory while dogfooding existing workflow commands.

Add:

- `scripts/service-launch <service> [--auto-confirm]`
- `just service-launch service=<service> auto_confirm=false`

The workflow resolves the named Service, finds its `backend.vm`, then composes existing commands in this order:

1. `scripts/vm-up <backend-vm>` with `--auto-confirm` only when requested.
2. `scripts/service-deploy <service>`.
3. `scripts/ingress-regenerate` only when the Service declares `ingress.enabled: true`.

Service Launch must remain a wrapper. It must not implement its own VM readiness state machine, deploy engine, Host configuration flow, NAS reconciliation flow, or Ingress/DNS infrastructure launch flow.

## Acceptance criteria

- [x] `just service-launch service=<name>` invokes `scripts/service-launch <name>`.
- [x] `auto_confirm=true` on the Just target passes `--auto-confirm` through to `scripts/service-launch`, and the script passes it through only to `scripts/vm-up`.
- [x] The script validates that the named Service exists before running any workflow command.
- [x] The script validates that the Service has a declared `backend.vm` and that the Backend VM exists in Inventory before running any workflow command.
- [x] The script runs `scripts/vm-up <backend-vm>` before every Service Deploy and relies on `vm-up` to decide whether the VM is already ready.
- [x] The script runs `scripts/service-deploy <service>` after successful VM Lifecycle Convergence.
- [x] The script runs `scripts/ingress-regenerate` after successful Service Deploy only when the Service declares `ingress.enabled: true`.
- [x] The script does not run `host-bootstrap`, `host-configure`, `nas-reconcile*`, `service-deploy internal-ingress`, or Service Deploy for DNS Services as hidden prerequisites.
- [x] The script deploys only the named Service, even when other Services share the same Backend VM or Service Group.
- [x] If `vm-up`, `service-deploy`, or `ingress-regenerate` fails, the script stops immediately, exits non-zero, and reports the failed phase without rolling back or destroying the Backend VM.
- [x] Fast workflow tests cover command order, `auto_confirm` passthrough, invalid Service names, missing Backend VM declarations, missing Backend VM Inventory, Ingress-enabled versus non-Ingress Services, singular Service scope, and downstream failure stopping behavior.
- [x] Documentation or runbook text describes Service Launch as a wrapper over `vm-up`, `service-deploy`, and conditional `ingress-regenerate`, with Host, NAS, and Ingress infrastructure readiness as prerequisites.
