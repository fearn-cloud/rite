## Agent Brief

**Category:** enhancement
**Summary:** Add validated Quadlet Fragment sidecar support for native options that fortress does not model directly.

**Current behavior:**
Fortress can render validated Quadlet Services into deterministic rootful system Quadlet `.container` and `.network` artifacts. The renderer owns Service invariants such as generated runtime names, image references, network attachment, Published Ports, Volumes, generated dependencies, install behavior, and disabled auto-update behavior.

There is not yet a supported way for an operator to add native Quadlet or systemd options that fortress intentionally does not model. ADR-0016 says those options belong in validated Quadlet Fragment sidecar files, not in raw replacement Quadlet files or a broader Compose-like Service schema.

**Desired behavior:**
Services should support plaintext Quadlet Fragments under `inventory/services/<service>.quadlet.d/`. Container fragments should be named `<container>.container`, matching declared container names. A Service may also have an optional `network.network` fragment for the generated network artifact.

Fragments should use native Quadlet/systemd INI syntax and should be merged into the generated final Quadlet artifacts. Unknown fragment filenames must be invalid, including stale fragments for containers no longer declared by the Service yaml.

Fragment validation should protect fortress-owned invariants. A fragment must not override generated scalar ownership such as image, container identity, network, Published Ports, Volumes, secrets, install behavior, auto-update policy, or generated dependencies. Forbidden keys should be derived from the generated Quadlet artifact for the relevant Service/container so the validator follows the renderer's actual output rather than a separately hand-maintained list.

Repeated additive dependency keys such as `Unit.Requires`, `Unit.After`, and `Unit.Wants` may add extra values, but must not replace or remove fortress-generated dependencies. Fragment-provided values should merge with generated values deterministically.

Fragments are plaintext-only. They must not contain Service secret values. Service Secrets continue to flow only through the Service Sibling SOPS File and `_FILE` injection.

**Key interfaces:**
- Quadlet Service rendering API — should render final artifacts with any applicable validated fragments merged into the generated content.
- Quadlet Fragment discovery — should find fragments from `inventory/services/<service>.quadlet.d/`, associate each file with a declared container or the generated network, and reject unknown or stale filenames.
- Quadlet/systemd INI parser or representation — should preserve native section/key semantics well enough to validate and merge repeated keys deterministically.
- Fragment validation — should compare fragment keys against the generated artifact to prevent overrides of fortress-owned scalar keys while allowing approved additive keys.
- Golden-file tests — should demonstrate successful merge behavior and rejection cases for invalid fragments.

**Acceptance criteria:**
- [x] Quadlet Fragments are discovered under `inventory/services/<service>.quadlet.d/`.
- [x] Container fragments are named `<container>.container`; the optional network fragment is named `network.network`.
- [x] Unknown fragment filenames are rejected, including stale fragments for containers no longer declared by the Service yaml.
- [x] Fragments use native Quadlet/systemd INI syntax and are merged into the generated final Quadlet files.
- [x] Fragment validation derives forbidden keys from the generated Quadlet for that Service/container so generated scalar ownership cannot be overridden.
- [x] Repeated additive keys such as `Unit.Requires`, `Unit.After`, and `Unit.Wants` can add values without replacing or removing fortress-generated dependencies.
- [x] Fragments cannot override fortress-owned image, container identity, network, Published Ports, Volumes, secrets, install behavior, auto-update policy, or generated dependencies.
- [x] Fragments are plaintext-only and cannot contain Service secret values; Service Secrets still go through the Service Sibling SOPS File and `_FILE` injection.
- [x] Golden-file tests cover successful fragment merging, unknown filename rejection, forbidden key rejection, and additive dependency merging.

## Implementation Notes

- Added Quadlet Fragment discovery to the Quadlet renderer via `inventory_root`, with fragments loaded from `inventory/services/<service>.quadlet.d/`.
- Added native INI parsing, generated-key ownership validation, reserved install/update/secret guards, and additive `Unit.Requires`/`Unit.After`/`Unit.Wants` merging.
- Added golden and behavior tests for merge, unknown/stale filenames, invalid INI, forbidden generated keys, reserved keys, and additive dependency preservation.

**Out of scope:**
- Changing the issue-07 Service schema beyond the minimum needed to wire fragment discovery into validation/rendering.
- Raw replacement Quadlet files.
- Rootless Quadlets.
- Service Secret injection implementation; this belongs to the dedicated Service Secret issue.
- The full Service deploy workflow beyond consuming the final rendered Quadlet artifacts.
