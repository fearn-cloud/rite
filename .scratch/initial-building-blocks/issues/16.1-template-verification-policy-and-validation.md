# Template Verification Policy and validation

## Parent

.scratch/initial-building-blocks/issues/16-template-verification-workflow.md

## Agent Brief

**Category:** enhancement
**Summary:** Add the Template Verification Policy shape and validation rules needed before temporary Template Verification VMs can be generated.

**Current behavior:**
Fortress has Host, VM, Service, and Template inventory, but it does not have a checked-in Template Verification Policy. The VM schema does not distinguish ordinary VMs from Operational VMs, and cross-file validation does not yet enforce the VMID ranges or reserve the `tmp-` VM name prefix for generated temporary declarations.

**Desired behavior:**
The repo should have a fleet-level Template Verification Policy that declares the reusable Operational VM slot for Template verification. VM inventory should be able to mark a VM as Operational, and validation should enforce that ordinary VMs stay in the ordinary VMID range while Operational VMs may use the operational range. Checked-in ordinary Inventory must not be allowed to use names beginning with `tmp-`.

The policy should not provision anything by itself. It is source data for later workflow slices.

**Key interfaces:**
- Template Verification Policy inventory shape — declares VMID `8901`, hardware cores and memory, disk size, per-Host storage, and per-Host address.
- VM inventory lifecycle marker — supports Operational VMs using `lifecycle.kind: operational`, plus purpose and generated metadata for the Template Verification VM.
- VM schema and cross-file validation — enforce ordinary, operational, and Template VMID range discipline and reserve checked-in `tmp-` names.
- Git ignore rules — ignore generated temporary Template Verification VM yaml and Sibling SOPS File artifacts.

**Acceptance criteria:**
- [ ] A checked-in Template Verification Policy exists and declares VMID `8901`.
- [ ] The policy declares hardware cores, memory, disk size, `storage_by_host`, and `address_by_host`.
- [ ] The policy has schema coverage or equivalent validation so missing required fields fail clearly.
- [ ] VM inventory supports `lifecycle.kind: operational`, `lifecycle.purpose`, and `lifecycle.generated`.
- [ ] Ordinary VMs using VMID `8900`-`8999` fail validation unless marked as Operational VMs.
- [ ] Operational VMs using VMIDs outside `8900`-`8999` fail validation.
- [ ] Template VMIDs remain reserved for Templates, not VM inventory.
- [ ] Checked-in ordinary VM inventory names beginning with `tmp-` fail validation.
- [ ] Generated temporary Template Verification VM artifacts are ignored by git.
- [ ] Tests cover valid policy inventory, missing policy fields, ordinary VMID range violations, Operational VMID range violations, and reserved `tmp-` names.

**Out of scope:**
- Generating `tmp-template-verify.yaml`.
- Running `qm config` against a Host.
- Adding `just template-verify`.
- Running Ansible verification inside a VM.
