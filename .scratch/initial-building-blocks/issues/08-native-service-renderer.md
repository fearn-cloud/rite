## What to build

The escape hatch for services genuinely better as native packages (Caddy is the canonical case). Same service yaml schema, different deploy block. Apt repo handling, config-file templating with reload-vs-restart logic.

- [ ] Reload-vs-restart logic respected: reload-flagged template change triggers `systemctl reload`; restart-flagged triggers `systemctl restart`
- [ ] Multi-config-file services supported

## Agent Brief

**Category:** enhancement
**Summary:** Add the Native Service renderer and deploy path for apt-backed systemd units.

**Current behavior:**
Services are declared in Inventory and deployed through `just service-deploy`, with the implemented deploy path centered on Quadlet Services. The Service schema already recognizes a Native deploy branch in minimal form, and the architecture/ADRs establish Native as the escape-hatch substrate for Services that are better installed as apt packages, with Caddy as the canonical example. The native renderer, deploy variables, playbook behavior, and tests are not yet complete.

**Desired behavior:**
A Service with `deploy.type: native` should be deployable through the same operator surface as Quadlet Services. Native deployment should install the declared apt package, optionally configure a referenced apt repository from Inventory globals, render one or more declared config files, and manage the declared systemd unit. Config changes must trigger reload or restart according to each config file's declaration: reload-flagged changes should reload the unit, restart-flagged changes should restart it, and a restart should take precedence when both kinds of files change in the same deploy. Multiple config files for the same Native Service should be supported.

**Key interfaces:**
- Service Inventory schema for `deploy.type: native` — should define the package name, optional apt repository reference, systemd unit name, and config-file declarations with source template, destination path, mode, and reload-vs-restart behavior.
- Native Service rendering/deploy interface — should produce the Ansible variables or rendered artifacts needed for the deploy playbook without requiring the Quadlet container fields.
- `service-deploy` operator workflow — should route Native Services to Native deployment behavior while preserving the existing Quadlet behavior.
- Apt repository references — should resolve against the existing Inventory globals convention for apt repositories and fail validation when a Service references a missing repository.
- Tests around the Native renderer/deploy contract — should assert rendered behavior for apt-repo handling, reload vs restart decisions, and multiple config files.

**Acceptance criteria:**
- [ ] Service Inventory validation accepts well-formed `deploy.type: native` Services and rejects missing package names, missing systemd unit names, invalid config-file declarations, and missing referenced apt repositories.
- [ ] `just service-deploy <service>` deploys a Native Service without trying to render Quadlet artifacts or install Podman-only resources.
- [ ] Native deployment installs the declared apt package and configures the optional referenced apt repository before package installation.
- [ ] Native deployment renders all declared config files with the declared destination paths and modes.
- [ ] When only reload-flagged config files change, deployment runs `systemctl reload` for the declared unit.
- [ ] When any restart-flagged config file changes, deployment runs `systemctl restart` for the declared unit, including when reload-flagged files also changed.
- [ ] Multi-config-file Native Services are supported in schema validation, rendering, and deployment.
- [ ] Golden-file or equivalent focused tests cover apt-repo handling, reload-vs-restart behavior, and multi-config-file rendering.
- [ ] A demo Native Service can be deployed through `just service-deploy`.

**Out of scope:**
- Replacing or changing the Quadlet deploy behavior completed in the previous issue.
- Native Service deletion/destruction workflows.
- General host package upgrades or apt dist-upgrade behavior.
- Redesigning Ingress generation or DNS regeneration beyond what is needed for a Native Service to be deployable.
