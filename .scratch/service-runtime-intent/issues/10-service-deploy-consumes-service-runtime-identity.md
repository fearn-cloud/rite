# Service Deploy consumes Service Runtime Identity

Status: ready-for-agent
Category: enhancement
Date: 2026-05-19

## Parent

`.scratch/service-runtime-intent/PRD.md`

## What to build

Migrate Service Deploy helpers and `scripts/service-deploy` so deploy preparation consumes **Service Runtime Intent** facts for **Service Runtime Identity** wherever a runtime result is available. Service Deploy should continue to build the same Ansible variables and run the same playbook, but the source of runtime names, paths, secret install specs, start/stop units, and share-backed subpaths should be Service Runtime Intent rather than raw Service yaml traversal.

Keep transitional fallback traversal for now. The final issue removes it after human review.

## Acceptance criteria

- [ ] `scripts/service-deploy` derives Backend VM, Service Secret requirements, Native Service Environment Secret requirements, and deploy helper inputs from the per-Service Service Runtime Intent view where practical.
- [ ] `quadlet_deploy_vars` consumes Service Runtime Intent facts for Service Data Directories, Service-owned paths, Service Network units, start units, stop units, Service Secret install specs, and share-backed subpaths when runtime intent is provided.
- [ ] `native_deploy_vars` continues to consume Native Service Environment Secret runtime facts.
- [ ] Existing Ansible variable names and values remain compatible unless a deliberate change is documented in this issue.
- [ ] Service Deploy still refuses missing VM Sibling SOPS Files and missing Service Sibling SOPS Files at the same operator-facing points.
- [ ] Service Deploy tests assert external behavior through `--extra-vars` while direct Service Runtime Intent tests own the deeper runtime identity assertions.
- [ ] Transitional fallback remains available for helper tests or call sites that do not yet pass Service Runtime Intent.
