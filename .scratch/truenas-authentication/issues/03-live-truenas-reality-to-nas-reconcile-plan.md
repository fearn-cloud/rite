## Agent Brief

**Category:** enhancement
**Summary:** Complete live TrueNAS reality mapping into the existing NAS Reconcile Plan contract, especially Fortress-owned NFS Share identity and marker handling.

**Current behavior:**
Live NAS Reconcile can already run the selected NAS Endpoint through the TrueNAS client adapter after preflight and map live responses into a `NasReality`-compatible payload. Dataset mapping includes the Dataset mount path and root owner UID/GID from live filesystem metadata. NFS Share mapping includes path, read/write access, clients, and a name derived from the TrueNAS response.

The remaining gap is that live NFS Share mapping does not clearly preserve the durable Fortress ownership marker contract used by existing planning logic. The planner recognizes Fortress-owned Shares by the durable marker shape `fortress:nfs-share:<share-name>` or by an explicit internal ownership flag. Live mapping currently risks treating a TrueNAS display/comment field as both the Share name and marker, which can prevent stale, drifted, and overlapping Share detection from matching fixture-backed behavior.

**Desired behavior:**
After live preflight succeeds, `scripts/nas-reconcile-plan --live <endpoint>` should build the same read-only NAS Reconcile Plan shape as fixture-backed `--reality-json` mode, with live TrueNAS reality mapped into the existing `NasReality` contract. Dataset validation and Derived Share planning should work unchanged whether reality came from a fixture or from live TrueNAS.

Live Dataset reality should include enough metadata to validate declared Dataset paths and root owner UID/GID. Live NFS Share reality should include enough metadata to detect missing desired Shares, stale Fortress-owned Shares, drifted Fortress-owned Shares, and unmanaged overlapping Shares. Fortress-owned Share recognition must use the same durable ownership marker semantics as fixture-backed planning.

Plan output should continue to redact connection information and report live credentials through the operator-environment credential source, not by printing credential values. Existing fixture-backed behavior should remain unchanged.
