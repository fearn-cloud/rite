## What to build

Extend Host shell access so the Operator can run a non-interactive command on a Host through the same decrypted per-Host credential path used for interactive Host shells.

The operator-facing behavior should mirror VM shell command execution:

```sh
scripts/host-shell <host> -- <command>...
```

This gives Host Readiness a reusable reachability proof such as `scripts/host-shell <host> -- true`, without duplicating Host SSH key decryption, Ansible Inventory connection variable resolution, or SSH argument construction.

## Acceptance criteria

- [ ] `scripts/host-shell <host>` still opens an interactive SSH session as it does today.
- [ ] `scripts/host-shell <host> -- <command>...` runs the remote command without allocating an interactive TTY.
- [ ] The command form continues to load Host connection details from Ansible Inventory.
- [ ] The command form continues to run through `scripts/decrypt-keys` with the Host Sibling SOPS File.
- [ ] The command exits with the remote command's exit code.
- [ ] Invalid argument shapes print usage and exit nonzero before running `ansible-inventory`, `decrypt-keys`, or `ssh`.
- [ ] Existing `host=<name>` compatibility remains for the Host argument.
- [ ] Fast tests cover interactive behavior, command behavior, remote exit-code propagation, and usage rejection.

## Context

- `scripts/vm-shell` already supports `-- <command>...`; Host shell should use the same operator-facing shape.
