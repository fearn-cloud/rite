## Agent Brief

**Category:** enhancement
**Summary:** Build declared Proxmox VM Templates from checksum-pinned Cloud Images.

**Current behavior:**
The repository can declare Template recipes under `inventory/templates/`, and Hosts can list which Templates they should hold under `proxmox.templates`. The Template schema requires a name, a 9000-9999 VMID, a source URL with SHA-512 checksum, supported `virt-customize` operations, and hardware defaults. The operator command `just templates-build host=<name>` invokes `scripts/templates-build`, which downloads and verifies Cloud Images, customizes working copies, and creates Proxmox Templates through fake-tool-covered wrapper tests. The remaining acceptance item is a real-Host demo on `wintermute`.

**Desired behavior:**
The operator can run `just templates-build host=<name>` to build every Template listed by that Host declaration. For each declared Template, the workflow downloads the source Cloud Image into a checksum-addressed cache, verifies the declared SHA-512 checksum before use, copies the cached image into a per-build working file, applies supported `virt-customize` operations to the working copy, creates a Proxmox VM at the declared Template VMID, attaches/imports the disk, configures hardware defaults and a cloud-init drive, then marks the VM as a Template.

The workflow is idempotent. If the VMID already exists on the target Host and is already a Template, skip it without changing it. If the VMID exists but is not a Template, fail with a clear error and do not delete, overwrite, or convert that VM. Force rebuilds and destructive replacement are out of scope for this issue.

**Key interfaces:**
- Template YAML under `inventory/templates/<name>.yaml` is the Template recipe source of truth.
- Host YAML `proxmox.templates` is the Host-to-Template assignment list.
- Template source checksum uses SHA-512 and is required before any Cloud Image is accepted.
- Template hardware defaults should include storage and device defaults needed by the builder. Unless overridden, Template disk storage defaults to `fast`, cloud-init drive storage defaults to `local-lvm`, and network bridge defaults to `vmbr0`.
- The supported `customize` shape for this slice is `packages` and `run_commands`; unsupported customize keys should either be rejected by schema or fail validation before touching a Host.
- The builder should use standard Proxmox `qm` operations for Cloud-Init Templates: create a VM, attach/import the Cloud Image as a SCSI disk, add a cloud-init drive, set boot/serial options when requested by hardware defaults, and convert with `qm template`.

**Acceptance criteria:**
- [x] `just templates-build host=<name>` invokes a real workflow instead of the current stub and rejects undeclared Host names.
- [x] The workflow builds only Templates listed under the target Host's `proxmox.templates`.
- [x] Template schema and fixtures cover explicit storage/network defaults, including default disk storage `fast`, cloud-init storage `local-lvm`, and default bridge `vmbr0`.
- [x] Template schema or validation accepts only the supported `customize.packages` and `customize.run_commands` operations for this slice.
- [x] The builder downloads Cloud Images into a checksum-addressed cache and verifies the declared SHA-512 checksum before using a cached or newly downloaded image.
- [x] Checksum mismatch fails before `virt-customize` or any `qm` mutation runs.
- [x] `virt-customize` runs only against a per-build working copy, never directly against the cached Cloud Image.
- [x] For a missing Template VMID, the workflow creates a Proxmox VM at the declared VMID, imports or attaches the customized disk, sets hardware defaults, adds the cloud-init drive, configures boot/serial options as declared, and marks the VM as a Template.
- [x] If the declared VMID already exists and is already a Template, the workflow skips it and a second run is a no-op.
- [x] If the declared VMID already exists but is not a Template, the workflow fails clearly without deleting, overwriting, or converting it.
- [x] Unit or wrapper tests use fake tools to cover command validation, checksum failure, cache hit behavior, working-copy customization, skip-on-existing-template, and fail-on-existing-non-template behavior.
- [x] Ansible syntax or equivalent workflow validation passes for any new playbook/role.
- [x] `runbooks/new-template.md` documents adding a Template recipe, assigning it to a Host, running `just templates-build`, checksum update expectations, cache behavior, idempotency, and the non-destructive VMID collision rule.

- No Ansible playbook or role was added for this slice, so the equivalent workflow validation is the wrapper test suite.

**Out of scope:**
- Force rebuilding or replacing an existing Template.
- Deleting or converting an existing non-Template VM at the declared VMID.
- Building VM lifecycle or OpenTofu clone resources.
- Supporting arbitrary `virt-customize` operations beyond `packages` and `run_commands`.
- Managing Host storage creation or registration; Host Configure continues to only document storage.
