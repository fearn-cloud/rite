## What to build

Build a first-class NFS shared-mount Acceptance Test workflow that generates two temporary Operational VMs on one Host from the same Template, reconciles one shared Derived NFS Share for the existing `acceptance-nfs-demo` Ephemeral Dataset, provisions both VMs through ordinary `vm-up`, and proves bidirectional create/read/delete behavior over the mounted Share through SSH.

The operator-facing command is:

```sh
just acceptance-nfs-shared-mount host=<host> template=<template> endpoint=<nas-endpoint> auto_confirm=false keep_on_fail=false
```

It delegates to:

```sh
scripts/acceptance-nfs-shared-mount host=<host> template=<template> endpoint=<nas-endpoint> auto_confirm=<true|false> keep_on_fail=<true|false>
```

The workflow should require `host`, `template`, and `endpoint`; no lab defaults are hidden. Without `auto_confirm=true`, it should print the full acceptance intent and require one workflow-level confirmation. With `auto_confirm=true`, it should skip that prompt and call `vm-up --auto-confirm` internally for both generated VMs.

## Notes

- Do not create an ADR for this workflow. It follows existing decisions: Acceptance Tests may create disposable Operational VMs, Ephemeral Datasets are acceptance-only, Shares are derived from Mount intent, NFS Mounts are systemd units, and `vm-up` is the operator-facing VM lifecycle path.
- Root-level SSH filesystem checks are intentional for v1. They prove the shared Mount contract without settling Dataset ACL, mode, or mapped UID/GID semantics.
- Cleanup should destroy VM clients before NAS resources to avoid stale active NFS mounts while removing the Share or Dataset.
