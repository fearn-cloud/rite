# Operator command for generated Acceptance file cleanup

Status: done
Category: enhancement

## What to build

Add an operator-facing command that removes generated Acceptance Test inventory artifacts without requiring the Operator to know every temporary file path by hand.

This command is for generated repository files left behind after Acceptance Test failures, `keep_on_fail=true` runs, interrupted workflows, or future Acceptance Test changes that add more temporary artifacts.

## Agent Brief

**Category:** enhancement
**Summary:** Add an operator command that cleans generated Acceptance Test files without requiring the Operator to know every temporary artifact path.

**Current behavior:**
Acceptance Test workflows generate temporary repository artifacts such as Operational VM declarations, VM Sibling SOPS Files, generated Service inventory, Service Sibling SOPS Files, and Quadlet Fragment directories. A separate issue will make Ephemeral Dataset YAML declarations generated during Acceptance Tests as well.

When generated files are preserved after `keep_on_fail=true`, workflow failure, or interruption, the Operator currently has to know which paths belong to each Acceptance Test and delete them manually. That is fragile, especially as the set of generated artifacts changes.

**Desired behavior:**
The Operator should have a single generated Acceptance artifact cleanup command exposed through `just` and a script entrypoint. The command should understand existing Acceptance Test workflows and remove only repository files or directories that are clearly generated Acceptance artifacts.

The Operator should be able to run the command for a specific Acceptance Test workflow and, if appropriate, for all known Acceptance Test workflows. The command should derive the expected generated paths from Acceptance Policy data and generated-artifact conventions so the Operator does not need to remember VM names, Service names, Dataset filenames, Sibling SOPS Files, or Quadlet Fragment paths.

By default, the command should behave conservatively: show the files and directories it intends to delete, require confirmation, tolerate already-missing files, and refuse anything that is not clearly generated and acceptance-owned. An explicit auto-confirm option should support scripted cleanup.

**Key interfaces:**
- Operator command surface — add a `just` recipe and script entrypoint for generated Acceptance artifact cleanup.
- Acceptance Policy inventory — use workflow-specific policy data such as generated VM identities and Dataset identity to derive expected generated artifact paths.
- Generated artifact conventions — reuse the existing conventions for temporary Acceptance VM, Service, Sibling SOPS, Quadlet Fragment, and Ephemeral Dataset artifacts.
- Cleanup reporting — print a concise plan before confirmation and a concise result summary after deletion.
- Safety checks — distinguish generated Acceptance artifacts from ordinary Inventory, and refuse ambiguous or unmanaged paths.
