## What to build

Add the supported OpenTofu wrapper entry point. The wrapper decrypts current Host PVE API token values from Host Sibling SOPS Files into ephemeral `TF_VAR_pve_token_<host>` environment variables, runs tofu from the `tofu/` working directory, and cleans up after itself. Tofu must never read SOPS directly.

## Acceptance criteria

- [x] A script such as `scripts/tofu-wrap` is the supported tofu entry point.
- [x] The wrapper reads Host declarations from `inventory/hosts/*.yaml`.
- [x] For each Host, the wrapper decrypts only `pve_tokens.tofu.value` from `inventory/hosts/<host>.sops.yaml`.
- [x] The wrapper exports only the matching `TF_VAR_pve_token_<host>` environment variable for each Host token.
- [x] The wrapper invokes `tofu` with its working directory set to `tofu/`.
- [x] The wrapper runs provider generation before tofu commands that require generated HCL.
- [x] Missing Host Sibling SOPS Files or missing tofu token values fail clearly before invoking tofu.
- [x] Token values are not printed to stdout/stderr by the wrapper.
- [x] Temporary files or env state are cleaned up on exit.
- [x] Tests use fake `sops` and fake `tofu` to verify decrypt extracts, env var names, working directory, command forwarding, and cleanup behavior.
- [x] Tests verify the wrapper does not decrypt entire SOPS files when an extract path is sufficient.
- [x] Documentation states that direct tofu invocation is unsupported because tofu must never read SOPS.

## Design decisions

- Secret transport is split from real VM resource creation.
- PVE tokens are ephemeral process environment only.
- Sensitive variables are declared by the generated/root tofu configuration; the wrapper owns only decryption and invocation.

## Out of scope

- PVE token creation or rotation.
- Host Configure changes.
- Real Proxmox VM resources.
- VM lifecycle orchestration.
