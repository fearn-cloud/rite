# Cloud-init does not apply VM admin user and SSH key for wintermute-demo

## Report

When running `just vm-up vm=wintermute-demo`, cloud-init does not appear to contain or apply the default VM admin user or the SSH public key needed for Ansible automation.

## Context

Fortress expects VM bring-up to follow Prepare -> OpenTofu apply -> Configure. Prepare writes the VM-specific public SSH key into the VM YAML. OpenTofu uploads per-VM cloud-init userdata. Cloud-init should create the default VM admin user from global inventory, authorize the prepared public key, set hostname/network intent, and leave the VM reachable so Configure can continue through Ansible.

This is a bug against the VM lifecycle contract, not a new feature. It touches the cloud-init first-boot boundary described by ADR-0010 and the OpenTofu/Ansible ownership boundary described by ADR-0002.

## What we established so far

- `inventory/vms/wintermute-demo.yaml` exists and includes `ssh_public_key`.
- `inventory/group_vars/all.yaml` sets `vm_admin_user: admin`.
- The OpenTofu VM partition module currently renders cloud-init userdata with `users`, `name: ${var.admin_user}`, `sudo`, and `ssh_authorized_keys`.
- `vm-configure` waits for SSH readiness and then `cloud-init status --wait`, so if the admin user or key is absent, Configure either cannot connect or is only succeeding through some unintended path.
- The completed wintermute demo issue recorded a successful `just vm-up vm=wintermute-demo` run on 2026-05-04, including a prior change from cloud-init user `fortress` to `admin`, so this may be a regression, stale Proxmox snippet/application state, or a false-positive configure path.
