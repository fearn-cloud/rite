# VM Mounts reference Datasets

Status: ready-for-agent
Category: enhancement

## What to build

Replace the old `nfs_mounts[].export` contract with VM `mounts[]` declarations that reference Datasets by name. Each Mount uses flat fields: `name`, `dataset`, `protocol`, `mount_point`, and `access`. Mount Names are VM-local, `protocol` is required, and `access` is `read_only | read_write`.

## Acceptance criteria

- [ ] `inventory/vms/_schema.json` accepts `mounts[]` with required `name`, `dataset`, `protocol`, `mount_point`, and `access`, and no longer requires or documents `nfs_mounts[].export`.
- [ ] Cross-file validation rejects Mounts referencing missing Datasets.
- [ ] Cross-file validation rejects duplicate Mount Names within a VM.
- [ ] Cross-file validation rejects Mount-bearing VMs without an unambiguous static IP address.
- [ ] The VM NFS mount Ansible role renders systemd `.mount` units from `mounts[]` where `protocol: nfs`.
- [ ] `access` is authoritative: `read_only` renders read-only behavior, and protocol options that contradict `access` are rejected.

## Agent Brief

**Category:** enhancement
**Summary:** Replace legacy VM NFS export declarations with Dataset-backed VM Mount declarations.

**Current behavior:**
VM inventory still models NAS access through legacy NFS mount declarations that name an export directly. The VM schema accepts that legacy shape, existing validation cannot fully reason about VM Mount references to first-class Datasets, and the VM NFS mount role renders systemd `.mount` units from the old contract.

First-class Dataset inventory and validation already exist, so this issue can build on Dataset names as stable Inventory references. The storage domain model says Datasets are durable data, Shares are derived access surfaces, and a VM Mount is the VM-side systemd `.mount` unit that references a Dataset through an explicit protocol and access policy.

**Desired behavior:**
VM inventory should declare optional `mounts` entries. Each Mount should have a VM-local `name`, a `dataset` reference, required `protocol`, `mount_point`, and `access`. `access` should be limited to `read_only` or `read_write` and should be the authoritative source for whether the resulting mount is read-only or read-write.

Validation should reject Mounts that reference missing Datasets, duplicate Mount Names within the same VM, and Mount-bearing VMs that do not have exactly one unambiguous static IP address usable for derived Share client access. NFS Mount rendering should consume `mounts` entries where `protocol` is `nfs` and continue rendering systemd `.mount` units rather than fstab, autofs, or in-container mounts.

Contradictory protocol options must not be allowed to override `access`. For example, a `read_only` Mount must render read-only NFS behavior, and a `read_write` Mount must not accept protocol options that force read-only behavior.

**Key interfaces:**
- VM inventory schema should accept the new `mounts` collection and stop requiring or documenting the old `nfs_mounts[].export` contract.
- Inventory loading and cross-file validation should expose VM Mount declarations and validate Dataset references, VM-local Mount Name uniqueness, and static-IP requirements.
- VM Configure host variables should present NFS Mounts to the Ansible role in a shape derived from Dataset-backed Mount declarations.
- The VM NFS mount Ansible role should render systemd `.mount` units for Mount declarations whose protocol is `nfs`.
- Protocol option handling should make `access` authoritative and reject options that contradict it.

**Acceptance criteria:**
- [ ] VM schema validation accepts Mount declarations with required `name`, `dataset`, `protocol`, `mount_point`, and `access`.
- [ ] VM schema validation rejects incomplete Mount declarations and no longer requires or documents `nfs_mounts[].export`.
- [ ] Cross-file validation rejects Mounts whose `dataset` does not resolve to a declared Dataset.
- [ ] Cross-file validation rejects duplicate Mount Names within a single VM while allowing the same Mount Name on different VMs.
- [ ] Cross-file validation rejects a VM with one or more Mounts unless that VM has exactly one unambiguous static IP address.
- [ ] VM Configure renders systemd `.mount` units from `mounts` entries where `protocol` is `nfs`.
- [ ] `read_only` Mounts render read-only NFS behavior, and `read_write` Mounts render read-write behavior.
- [ ] Protocol options or extra mount options that contradict the declared `access` are rejected before rendering.
- [ ] Existing tests and fixtures are updated from the legacy NFS export contract to the Dataset-backed Mount contract where applicable.

**Out of scope:**
- Implementing NAS Reconcile plan or TrueNAS API writes.
- Creating, updating, or deleting NAS-side Shares.
- Implementing Share-backed Service volumes.
- Supporting non-NFS protocols beyond validating that `protocol` is present and only rendering NFS entries through the existing NFS mount role.
- Migrating live VMs or performing manual NAS changes.
