# Model ingress VM live host edits in inventory

## Agent Brief

**Category:** enhancement
**Summary:** Move the live ingress VM address and SSH listener edits into inventory and have Rite converge them.

**Current behavior:**
The staged Git-over-SSH ingress work depends on live edits that were applied directly on `internal-ingress-vm`: the VM has an additional `10.40.0.21/24` address on its infrastructure interface, and OpenSSH is constrained to the management address so Caddy can bind the Git SSH ingress address on port 22.

Those facts are documented in the completed implementation ticket, and the service-level TCP ingress route points at `10.40.0.21`, but the VM inventory and convergence workflows do not yet own the underlying VM address or host OpenSSH listener policy. A fresh provision, rebuild, or configuration run could therefore drift away from the state required by the staged Caddy layer4 changes.

Redundancy check:

- The staged Service TCP Ingress Route work models the Caddy route and generated DNS record.
- VM inventory currently models one interface address per VM interface and does not declare the additional ingress address as an owned VM fact.
- The Tofu/cloud-init network intent and VM Configure workflow should be checked because they are the existing Rite surfaces for VM network and host-level convergence.
- The Service Deploy workflow should not be the long-term owner of these VM-level facts; it only happened to trigger the live changes during implementation.

Prior rejection check:

- No matching `.out-of-scope/` rejection exists.

**Desired behavior:**
Inventory should declare all VM-level facts required for the Git SSH ingress listener, and Rite should converge them idempotently.

Specifically, `internal-ingress-vm` should have an inventory-owned secondary IPv4 address on the same infrastructure interface for Git SSH ingress, and the VM's OpenSSH server should be configured from inventory so management SSH listens only where intended and does not steal the Git SSH listener.

The convergence path should make the live system match inventory without relying on hand-written netplan or sshd drop-ins. Re-running the appropriate Rite command should preserve these facts, validate them, and be safe when the live system is already correct.

**Key interfaces:**

- VM network inventory — must be able to express a primary interface address plus one or more additional static IPv4 addresses on the same VM interface, without breaking existing callers that expect a primary VM address.
- VM network intent rendered for provisioning and configuration — must include the additional address so new or rebuilt VMs receive the same network shape.
- VM Configure or the appropriate Rite VM convergence command — must apply the declared network address and the declared OpenSSH listen policy to an already-provisioned VM.
- VM inventory validation — must reject ambiguous or invalid additional addresses and preserve existing invariants for features that require a single primary VM address.
- Ingress TCP route validation — should ensure a Service TCP Ingress Route listen address is actually owned by the Ingress VM inventory, so `10.40.0.21` cannot be an undeclared magic value.
- OpenSSH listener policy — must support constraining VM management SSH to the intended management address for this VM while leaving ordinary VMs unchanged.

**Acceptance criteria:**

- [ ] `internal-ingress-vm` inventory declares both `10.40.0.16/24` and `10.40.0.21/24` as owned VM addresses, with `10.40.0.16` remaining the primary management address.
- [ ] Inventory validation accepts the declared multi-address ingress VM and rejects a Service TCP Ingress Route whose listener address is not declared on the Ingress VM.
- [ ] Tofu/cloud-init generated network intent includes the additional address for fresh provisioning or rebuilds.
- [ ] The appropriate Rite VM convergence command applies the additional address idempotently to an existing VM.
- [ ] The appropriate Rite VM convergence command installs and validates the OpenSSH listener policy so OpenSSH binds the management address and does not bind the Git SSH ingress address.
- [ ] Re-running the convergence command on an already-correct `internal-ingress-vm` reports success without unnecessary disruptive changes.
- [ ] Existing HTTP ingress DNS and Caddy generation behavior remains unchanged except that the Git SSH TCP listener address is now sourced from declared VM inventory.
- [ ] Tests cover VM inventory parsing/validation, generated network intent, OpenSSH listener rendering or deployment, and the TCP ingress listener-address validation path.

**Out of scope:**

- Changing the accepted Git SSH route shape or the Caddy layer4 design.
- Moving management SSH to a different port.
- Proving repository-level Forgejo SSH authorization or push access.
- General multi-interface routing policy beyond the address and listener control required for this ingress VM.
