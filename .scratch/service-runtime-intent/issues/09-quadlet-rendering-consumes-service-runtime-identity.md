# Quadlet rendering consumes Service Runtime Identity

Status: done
Category: enhancement
Date: 2026-05-19

## Parent

`.scratch/service-runtime-intent/PRD.md`

## What to build

Migrate the Quadlet Adapter so it consumes **Service Runtime Intent** facts for **Service Runtime Identity** wherever a runtime result is available. Quadlet should still render `.network` and `.container` artifacts, merge Quadlet Fragments, and enforce Quadlet-owned reserved-key behavior. It should stop independently deciding fortress-owned runtime names and paths when the resolved facts are present.

Keep transitional fallback traversal for now so existing helper-level tests and older call sites continue to work. The fallback is removed only by the final human-in-the-loop issue.

## Acceptance criteria

- [x] `render_quadlet_service` and `render_quadlet_container` consume Service Runtime Intent facts for Service Network names, Podman container names, Container Alias values, systemd unit names, Service-owned volume paths, Share-backed Volume paths, mount unit requirements, and Service Secret names when runtime intent is provided.
- [x] Quadlet rendering still owns artifact formatting: `.network` filenames/content, `.container` filenames/content, `Network=`, `Volume=`, `Secret=`, `Environment=`, `Requires=`, `After=`, `BindsTo=`, and fragment merging.
- [x] Rendered Quadlet output remains byte-for-byte compatible with existing golden fixtures unless a deliberate fixture update is documented in this issue.
- [x] Tests prove Quadlet rendering uses supplied Service Runtime Intent facts for at least one value that differs from what raw Service yaml fallback would derive.
- [x] Existing golden rendering tests continue to pass through the transitional fallback path where they do not yet build a fleet-wide Service Runtime Intent.
- [x] No SOPS decryption, Ansible variable construction, Service Deploy ceremony, Observability generated files, or Application Configuration Artifacts move into Quadlet rendering.
