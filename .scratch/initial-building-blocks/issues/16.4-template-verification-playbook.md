# Template verification playbook

## Parent

.scratch/initial-building-blocks/issues/16-template-verification-workflow.md

## Agent Brief

**Category:** enhancement
**Summary:** Add an Ansible verification playbook that proves a Template-backed VM satisfies the VM lifecycle contract from inside the Template Verification VM.

**Current behavior:**
The ordinary VM Configure workflow waits for cloud-init and configures VMs, but there is no dedicated verification playbook whose purpose is to assert that a Template-backed VM satisfies the minimum lifecycle contract before ordinary VMs depend on that Template.

**Desired behavior:**
After the Template Verification VM has been provisioned through the normal `vm-up` path, a verification playbook should connect to the running VM and assert the VM lifecycle contract from inside the VM. It should check that cloud-init completed, the configured VM admin user exists, passwordless sudo works, hostname is applied, and SSH key material agrees across the VM yaml, decrypted Sibling SOPS File, and the running VM's authorized key.

Failures should be specific enough for the orchestration wrapper to surface useful messages.

**Key interfaces:**
- Ansible verification playbook — runs against the generated Template Verification VM.
- VM lifecycle contract — cloud-init completion, configured VM admin user, authorized SSH public key, passwordless sudo, and hostname.
- VM yaml and Sibling SOPS File — provide the expected SSH public key values for comparison with the in-VM authorized key.
- Inventory plugin or equivalent hostvars — exposes the generated VM and secret material to the verification playbook without leaking private keys to persistent disk.

**Acceptance criteria:**
- [x] A Template verification playbook exists and can be invoked by the `template-verify` workflow.
- [x] The playbook asserts `cloud-init status --wait` completes successfully.
- [x] The playbook asserts the configured VM admin user exists.
- [x] The playbook asserts the configured VM admin user has passwordless sudo.
- [x] The playbook asserts the VM hostname matches the generated VM declaration.
- [x] The playbook asserts the VM yaml `ssh_public_key` and decrypted Sibling SOPS File `ssh_keys.bootstrap.public_key` agree.
- [x] The playbook asserts the in-VM authorized key contains the expected VM admin public key.
- [x] Verification failures identify which lifecycle contract check failed.
- [x] Tests or fake Ansible fixtures cover success plus failures for cloud-init, admin user, sudo, hostname, and key mismatch.

**Out of scope:**
- Generating the temporary VM declaration.
- Adding `host=all` orchestration.
- Destroying the Template Verification VM.
- Auto-building Templates.
