a# Bootstrap idempotency by refusal, not silent skip

`host-bootstrap.yml` refuses to run if the Host Sibling SOPS File already exists, while `vm-prepare.yml` refuses to run if the VM Sibling SOPS File already contains VM SSH key material. Bootstrap and Prepare are the only steps that mint irreversible SSH material; a silent skip inside those ceremonies would either overwrite a valid key or no-op past a real divergence. VM Prepare may merge generated SSH key material into a pre-existing VM Sibling SOPS File that contains non-SSH credentials, such as a Tailnet Auth Key, because those credentials must exist before Configure but do not mean the VM SSH key has already been minted.

Wrapper workflows may treat an already-completed ceremony as satisfied only after validating its stored outputs. In particular, VM Lifecycle Convergence may continue past Prepare when the VM Sibling SOPS File structurally contains VM private-key material and the public key recorded in Inventory matches the encrypted Sibling SOPS public key; mismatch remains a hard failure that points at explicit rotation or repair. Both public-key fields are required: `inventory/vms/<vm>.yaml:ssh_public_key` for OpenTofu/cloud-init wiring, and `inventory/vms/<vm>.sops.yaml:ssh_keys.bootstrap.public_key` as the credential record. If `ssh_keys.bootstrap.type` exists, it must decrypt to `vm_ssh`; missing `type` is tolerated for older credential blocks. `ssh_keys.bootstrap.created` is audit metadata and is not required for satisfaction. The existing private key is checked for presence without decrypting its value; private-key decryption belongs to VM access workflows such as VM Configure. This satisfaction check lives in VM Lifecycle Convergence, not inside `vm-prepare`, so `vm-prepare` remains the one-shot minting ceremony and the wrapper owns resumability. When Prepare is already satisfied, VM Lifecycle Convergence keeps a visible `prepare-satisfied` phase instead of silently omitting Prepare. The `prepare-satisfied` phase delegates to a dedicated probe script rather than inline plan-builder logic, because the check has its own domain diagnostics and is useful during direct debugging. A VM Sibling SOPS File without `ssh_keys.bootstrap` does not count as existing VM SSH material, because it may contain non-SSH credentials that Prepare is allowed to merge with. If `ssh_keys.bootstrap` exists and the probe fails, VM Lifecycle Convergence fails hard instead of falling back to `vm-prepare`; repairing or rotating VM identity is an explicit operator action.
