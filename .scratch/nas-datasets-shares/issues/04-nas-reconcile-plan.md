# NAS Reconcile Plan

Status: done
Category: enhancement

## What to build

Introduce a read-only NAS Reconcile Plan that compares declared Dataset and derived Share intent with TrueNAS reality without mutating TrueNAS. The plan validates adopted Dataset existence and root owner UID/GID, derives desired NFS Shares from VM Mount and Service consumption declarations, and reports unmanaged Shares that could expose the same Dataset.

## Acceptance criteria

- [x] A `nas-reconcile` or `nas-reconcile-plan` operator command loads inventory and produces a read-only plan.
- [x] TrueNAS connection settings and credentials are represented without exposing secrets in logs or plans.
- [x] The plan validates each adopted Dataset exists at its declared TrueNAS path.
- [x] The plan validates each adopted Dataset root owner UID/GID and reports drift without repairing it.
- [x] Desired NFS Shares are derived deterministically from Dataset, protocol, compatible Access Policies, and Mount-bearing VM static IP addresses.
- [x] Unmanaged Shares that could expose the same Dataset as desired fortress-owned Share intent block the plan.
- [x] Unit tests cover plan output with missing Dataset, owner drift, missing Share, stale fortress-owned Share, and unmanaged overlap.

## Agent Brief

**Category:** enhancement
**Summary:** Add a read-only NAS Reconcile Plan that compares declared Dataset and derived Share intent with TrueNAS reality.

**Current behavior:**
Inventory already models Datasets, VM Mounts, and Share-backed Service Volumes. Cross-file validation can prove Mounts reference declared Datasets, Mount Names are VM-local, Mount-bearing VMs have an unambiguous static IP address, and Services consume Mounts through Share-backed Volumes without widening access.

VM Configure can render NFS `.mount` units from Dataset-backed Mount declarations, but there is not yet an operator-facing NAS Reconcile Plan workflow that reads TrueNAS reality before VM Configure or Service deployment. The next issue will handle API-backed writes for fortress-owned Shares; this issue is the read-only planning phase.

**Desired behavior:**
The operator should have a `nas-reconcile` or `nas-reconcile-plan` command that loads Inventory, reads TrueNAS reality, and emits a deterministic plan without mutating TrueNAS.

The plan should validate each adopted Dataset exists at its declared TrueNAS path and that the Dataset root owner UID/GID matches the Inventory declaration. Missing Datasets and ownership drift should be reported as plan findings, not repaired.

The plan should derive desired NFS Shares from declared VM Mounts and Share-backed Service consumption. Desired Share identity must be deterministic from Dataset, protocol, and compatible Access Policies; it must not depend on Mount Name. NFS client access should be derived from Mount-bearing VM static IP addresses.

The plan should compare derived Share intent with existing TrueNAS Shares. Missing desired Shares, stale fortress-owned Shares, and unmanaged Shares that could expose the same Dataset as desired fortress-owned Share intent should be reported. Unmanaged overlap should block the plan until the Operator resolves it.

TrueNAS connection settings and credentials must be represented without exposing secrets in logs, errors, or plan output.

**Key interfaces:**
- Operator command for NAS Reconcile Plan — loads Inventory, reads TrueNAS state, and prints or returns a read-only plan.
- TrueNAS connection configuration — enough endpoint and credential information to read Dataset and NFS Share state while redacting secrets from user-visible output.
- NAS reality reader/client abstraction — exposes Dataset existence, Dataset root owner UID/GID, and current NFS Share declarations without requiring write support.
- Plan model/output — represents Dataset findings, desired Shares, missing Shares, stale fortress-owned Shares, unmanaged overlaps, and whether the plan is blocked.
- Desired Share derivation logic — groups compatible VM Mount access policies by Dataset/protocol and derives explicit NFS clients from VM static IP addresses.

**Acceptance criteria:**
- [x] A `nas-reconcile` or `nas-reconcile-plan` operator command loads Inventory and produces a read-only plan.
- [x] Running the plan does not create, update, delete, or repair anything in TrueNAS.
- [x] TrueNAS connection settings and credentials are represented without exposing secrets in logs, errors, or plan output.
- [x] The plan validates each adopted Dataset exists at its declared TrueNAS path.
- [x] The plan validates each adopted Dataset root owner UID/GID and reports drift without repairing it.
- [x] Desired NFS Shares are derived deterministically from Dataset, protocol, compatible Access Policies, and Mount-bearing VM static IP addresses.
- [x] Desired Share identity does not depend on VM-local Mount Name.
- [x] Missing desired Shares are reported in the plan.
- [x] Stale fortress-owned Shares are reported in the plan without being deleted.
- [x] Unmanaged Shares that could expose the same Dataset as desired fortress-owned Share intent block the plan.
- [x] Unit tests cover plan output with missing Dataset, owner drift, missing Share, stale fortress-owned Share, and unmanaged overlap.

**Out of scope:**
- Creating, updating, or deleting TrueNAS Shares.
- Repairing Dataset ownership or recursively changing Dataset contents.
- Creating, deleting, or adopting ordinary Datasets.
- Implementing durable fortress ownership markers beyond whatever read-only detection is needed for the plan.
- Operator confirmation flows for Mount removals, Mount access changes, or Mount point changes.
- Live Acceptance Tests against a real TrueNAS instance.
