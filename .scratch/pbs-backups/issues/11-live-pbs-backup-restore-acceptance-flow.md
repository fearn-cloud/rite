# Live PBS Backup Restore Acceptance Flow

## What to build

Create a live Acceptance Test flow that proves the PBS backup and restore path end to end using generated disposable Operational VMs. The flow should create a temporary Backup Target-like VM from declared acceptance intent, configure the fortress-owned Backup Job shape for it, trigger or wait for a Backup Run, restore the resulting restore point into an isolated disposable VM, verify a marker from VM-local state survived the PBS Restore, and clean up all generated live resources by default.

This is not a Restore Drill for an existing production Backup Target. It is a live infrastructure Acceptance Test that proves the backup and restore machinery itself works before production Restore Drills depend on it.

## Acceptance criteria

- [ ] A dedicated PBS backup/restore Acceptance Test can be invoked from a `just` recipe.
- [ ] The flow uses generated disposable Operational VMs and generated acceptance inventory rather than checked-in production VMs.
- [ ] The flow writes a deterministic VM-local marker before the Backup Run and verifies that marker after PBS Restore.
- [ ] The flow proves a fortress-owned Backup Job can be reconciled for the generated acceptance VM without weakening the production rule that generated/disposable VMs are exempt from ordinary backup declaration requirements.
- [ ] The flow can trigger or otherwise obtain a successful Backup Run for the generated acceptance VM and report when trigger submission is not yet proven backup protection.
- [ ] The flow restores the selected restore point into an isolated disposable restored VM without production identity, DNS, ingress, or NAS-backed Dataset mutation.
- [ ] Successful runs clean up generated inventory, Backup Jobs, restored VMs, source acceptance VMs, and other live resources by default.
- [ ] Failed runs clean up by default, with an explicit `keep_on_fail=true` path that preserves all generated artifacts needed for diagnosis and reports how to clean them.
- [ ] Operator output distinguishes the live Acceptance Test from Backup Readiness, Backup Health, and production Restore Drills.
- [ ] Tests cover planning, fake live client execution, marker verification, cleanup defaults, keep-on-fail preservation, and refusal to collide with production VM identity or production Backup Jobs.

## Agent Brief

**Category:** enhancement
**Summary:** Add a live PBS backup/restore Acceptance Test that proves a disposable VM can be backed up, restored, verified, and cleaned up.

**Current behavior:**
PBS backup intent, Backup Configure, Backup Readiness, Backup Health, and Restore Drill planning/execution are specified as separate operational workflows. Restore Drills prove recovery from backup reality for production Backup Targets. Existing Acceptance Tests can create disposable Operational VMs for live infrastructure contracts, but there is no dedicated Acceptance Test that exercises PBS backup and restore end to end against a generated disposable VM.

**Desired behavior:**
The operator should have a live PBS backup/restore Acceptance Test. It should use the Acceptance Test Lifecycle where appropriate for generated artifact writing, provisioning, failure preservation, and cleanup. The workflow should create a disposable source Operational VM, place a deterministic marker in VM-local state, reconcile or otherwise install the fortress-owned Backup Job needed for that generated VM, obtain a successful Backup Run, restore the resulting restore point into an isolated disposable restored VM, verify the marker exists in the restored VM, and then clean up generated artifacts by default.

The restored VM must be contained like a Restore Drill artifact: no production identity collision, no production DNS or ingress exposure, no production NAS-backed Dataset mutation, and operator-only access where restored secrets could exist. Output should be explicit that this proves the live backup/restore machinery, not production Backup Readiness, Backup Health freshness, or application-level Service health.

**Key interfaces:**
- Acceptance Policy for PBS backup/restore acceptance — should reserve the VMID, placement, storage, network, and IP allocation needed by this workflow without reusing unrelated acceptance slots.
- Acceptance Test Lifecycle integration — should own common generated Operational VM artifact creation, provisioning, failure preservation, and cleanup mechanics where those existing interfaces fit.
- Backup Configure or Backup Job reconciliation interface — should support the generated acceptance VM's temporary Backup Job without treating generated/disposable VMs as ordinary production Backup Targets.
- Backup Run trigger/query interface — should distinguish submitted backup tasks from confirmed successful restore points.
- PBS Restore or Restore Drill execution interface — should restore the selected acceptance restore point into an isolated disposable VM and expose enough verification hooks to inspect VM-local marker state.
- `just` recipe — should provide the operator entry point with `host`, `template`, and `keep_on_fail`-style controls consistent with other live Acceptance Tests.

**Acceptance criteria:**
- [ ] The operator can run a dedicated PBS backup/restore Acceptance Test from a stable `just` recipe.
- [ ] The workflow creates only generated disposable Operational VMs and generated acceptance inventory, never durable production VM inventory.
- [ ] The workflow writes and later verifies a deterministic VM-local marker across PBS Restore.
- [ ] The generated source VM can receive a temporary fortress-owned Backup Job without relaxing normal production backup validation.
- [ ] The workflow confirms a successful Backup Run/restore point before attempting PBS Restore.
- [ ] The restored VM is isolated from production VM identity, ingress, DNS, and production NAS-backed Dataset mutation.
- [ ] Success cleans up generated inventory, temporary Backup Jobs, source VMs, restored VMs, and other generated live resources.
- [ ] Failure cleans up by default, while `keep_on_fail=true` preserves generated artifacts and reports cleanup instructions.
- [ ] Tests cover fake-client execution and rendering for success, failed backup, failed restore, failed marker verification, cleanup, keep-on-fail, and production-collision refusal.

**Out of scope:**
- Application-specific restore validation for production Services.
- Changing the production Backup Policy model or the rule that generated/disposable VMs are exempt from ordinary production backup declarations.
- Off-site PBS replication, Backup Sync, or new Datastore design.
- PBS self-backup through the local PBS instance.
