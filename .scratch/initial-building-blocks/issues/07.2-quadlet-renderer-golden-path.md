## Agent Brief

> *This was generated by AI during triage.*

**Category:** enhancement
**Summary:** Render validated issue-07 Quadlet Services into deterministic rootful system Quadlet units with golden-file coverage.

**Current behavior:**
Fortress has the issue-07 Quadlet Service schema and validation contract in place, including singular Backends, Service Groups, Published Ports, image pinning, Service-owned Volumes, Share-backed Volumes, and Container Dependencies. The renderer is still a minimal scaffold: it can express some container volume behavior and Share-backed Volume mount ordering, but it does not yet produce the complete set of rootful system Quadlet `.container` and `.network` artifacts required by the Service contract.

**Desired behavior:**
Rendering a validated `deploy.type: quadlet` Service should produce deterministic rootful system Quadlet artifacts for each container and the Service's Podman network.

Runtime identities must be fortress-owned and service-scoped. Container unit names and Podman container names should use `fortress-<service>-<container>`. A Service without `service_group` should use an isolated network named `fortress-<service>`. A Service in a Service Group should use the shared network named `fortress-group-<service_group>`. The declared container `name` remains the Podman network DNS alias, while the rendered unit/container identity remains service-scoped.

Service-owned Volumes should bind from the Service Data Directory under `/srv/services/<service>/<service_path>` to the declared container path. When `storage.uid` and/or `storage.gid` are declared, renderer/deploy output should ensure ownership is applied only to Service-owned paths; Share-backed Volume ownership stays governed by the VM Mount and Dataset convention.

Share-backed Volumes should resolve their `mount` against the Backend VM's declared Mount. `source: /` binds the Mount root; any other `source` binds the already-validated relative subpath below that Mount. Containers consuming Share-backed Volumes must include `Requires=` and `After=` dependencies on the matching systemd `.mount` unit.

Container Dependencies from `depends_on` should render same-Service systemd start-order and stop-coupling between generated container units. This ordering must not claim or model application readiness.

The renderer should not emit Podman auto-update behavior. Image references are already validated for pinning; rendering should preserve those references exactly.

**Key interfaces:**
- Quadlet Service rendering API — should accept a validated Service plus its Backend VM context and return deterministic rendered `.container` and `.network` artifact content and names.
- Rootful system Quadlet artifact model — should represent generated filenames/unit names, container identity, network identity, unit dependencies, and file content in a way the deploy workflow can write under the system Quadlet location.
- Service-owned Volume rendering — maps `service_path` to `/srv/services/<service>/<service_path>` and maps the declared target/container path to the Quadlet volume destination.
- Share-backed Volume rendering — resolves VM Mount names to Mount paths, emits safe bind paths, and adds `.mount` unit dependencies.
- Container Dependency rendering — maps declared same-Service dependencies to generated container unit dependencies for start ordering and stop coupling.
- Golden-file test fixtures — should cover representative Services and expected rendered Quadlet output.

**Acceptance criteria:**
- [ ] Quadlet renderer produces rootful system `.container` and `.network` artifacts from the issue-07 Service schema.
- [ ] Rendered container unit names and Podman container names use `fortress-<service>-<container>`.
- [ ] Isolated Service networks render as `fortress-<service>`.
- [ ] Service Group networks render as `fortress-group-<service_group>`.
- [ ] Container aliases use the declared container `name` while rendered container identity remains service-scoped.
- [ ] Service-owned Volumes render from `/srv/services/<service>/<service_path>` to the declared container path.
- [ ] Declared Service Data Owner ownership is applied only to Service-owned paths, not Share-backed Volume paths.
- [ ] Share-backed Volumes render from the Backend VM Mount root for `source: /` and from validated relative subpaths for non-root sources.
- [ ] Share-backed Volumes add `Requires=` and `After=` dependencies on the matching `.mount` unit.
- [ ] `depends_on` renders same-Service start-order and stop-coupling systemd dependencies without modeling readiness.
- [ ] Auto-update behavior is absent from generated Quadlet output.
- [ ] Golden-file tests cover single-container, multi-container, isolated network, Service Group network, Service-owned volume, Share-backed root Mount volume, Share-backed subpath volume, Service Data Owner, dependency ordering, and image pinning variants.

**Out of scope:**
- Defining or changing the issue-07 Service schema and validator rules.
- Service Secret injection; that belongs to the dedicated Service Secret slice.
- Quadlet Fragment rendering or validation; that belongs to the dedicated Quadlet Fragment slice.
- The full `service-deploy` workflow, remote file writing, pruning, restart orchestration, and failure handling beyond producing renderer output that the deploy workflow can consume.
- Service deletion/destruction.
- Rootless Quadlets.
