## What to build

Document Host Readiness as the operator workflow for proving a new Proxmox Host is ready for ordinary VM and Service workflows.

The documentation should explain that Host Readiness is a resumable wrapper over existing commands, not a replacement for the lower-level ceremonies. It should describe when to use `host-up`, how endpoint scoping works, and what `auto_confirm` and `keep_on_fail` change.

## Acceptance criteria

- [ ] The new Host runbook documents `just host-up <host> endpoint=all auto_confirm=false keep_on_fail=false`.
- [ ] The runbook explains that `endpoint=all` runs acceptance against every declared NAS Endpoint.
- [ ] The runbook explains that `endpoint=<name>` scopes acceptance to one NAS Endpoint.
- [ ] The runbook explains that an already-completed Bootstrap is accepted only after the stored per-Host credential proves SSH reachability.
- [ ] The runbook explains that `host-up` runs full Host Configure, builds all Host-declared Templates, verifies all Host-declared Templates, and runs selected Acceptance Tests across Template x NAS Endpoint.
- [ ] The runbook explains that a Host with no declared Templates cannot pass Host Readiness.
- [ ] The runbook explains that `auto_confirm=true` makes supported downstream phases non-interactive.
- [ ] The runbook explains that `keep_on_fail=true` preserves generated artifacts where downstream workflows support preservation and may stop later cells to avoid artifact collisions.
- [ ] The documentation points readers to lower-level commands for surgical phase work rather than adding phase scoping to `host-up`.
- [ ] Tests that assert runbook coverage are updated or added.
