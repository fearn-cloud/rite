# Generate Acceptance Dataset inventory only during Acceptance Tests

Status: ready-for-human
Category: enhancement

## What to build

Change Acceptance Test workflows so their Ephemeral Dataset YAML declarations are generated as temporary acceptance-owned artifacts during the run, instead of being committed as ordinary `inventory/datasets/*.yaml` files.

The NFS shared-mount and Service-layer Acceptance Tests should still use deterministic Dataset names and paths, but those Dataset declarations should exist only while the Acceptance Test is running or while `keep_on_fail=true` preserves generated artifacts for inspection.

## Acceptance criteria

- [x] The NFS shared-mount Acceptance Test generates its Ephemeral Dataset declaration before any inventory load or validation phase that needs that Dataset.
- [x] The Service-layer Acceptance Test generates its Ephemeral Dataset declaration before any inventory load or validation phase that needs that Dataset.
- [x] Generated Dataset declarations include the same Dataset contract the workflows depend on: deterministic `name`, selected NAS Endpoint, deterministic acceptance path, and `lifecycle: ephemeral`.
- [x] Acceptance Policy inventory remains the durable place that names the workflow's Dataset identity and other reusable acceptance reservations.
- [x] The workflows refuse to overwrite a pre-existing Dataset YAML file at the generated path, just as they refuse to overwrite generated VM, Service, Sibling SOPS, and Quadlet Fragment artifacts.
- [x] Default cleanup removes generated Dataset YAML files after live resources are destroyed.
- [x] `keep_on_fail=true` preserves generated Dataset YAML files with the other generated acceptance artifacts for inspection.
- [x] Ordinary fleet validation no longer sees committed `acceptance-nfs-demo` or `acceptance-service-layer` Dataset declarations during normal operation.
- [x] Automated tests cover generated Dataset YAML contents, phase ordering before inventory-dependent operations, overwrite refusal, default cleanup, `keep_on_fail=true` preservation, and removal of durable acceptance Dataset files from ordinary inventory.

## Blocked by

None - can start immediately.

## Notes

- This issue narrows the Ephemeral Dataset model: Acceptance Policies can durably reserve Dataset identity, but concrete Ephemeral Dataset inventory should be generated only inside the Acceptance Test ceremony.
- Keep the existing safety posture from ADR 0018: ordinary Datasets are protected, Shares are derived, and only Acceptance Test Ephemeral Datasets may be created or destroyed.
- This should apply to both existing acceptance workflows that currently depend on committed Ephemeral Dataset declarations.

## Agent Brief

**Category:** enhancement
**Summary:** Generate Ephemeral Dataset YAML declarations during Acceptance Test runs instead of keeping them as committed ordinary inventory.

**Current behavior:**
The NFS shared-mount and Service-layer Acceptance Tests use dedicated Ephemeral Datasets. Their Acceptance Policies durably name the Dataset identities, and the concrete Dataset YAML declarations currently live in ordinary Dataset inventory. That means normal inventory scans can see acceptance-only Ephemeral Datasets even when no Acceptance Test is running.

The workflows already generate temporary Operational VM declarations and, for Service-layer acceptance, temporary Service artifacts. They also already have cleanup and `keep_on_fail=true` behavior for generated acceptance-owned artifacts.

**Desired behavior:**
Acceptance Tests should generate the concrete Ephemeral Dataset YAML declarations they need as part of the Acceptance Test ceremony. The generated Dataset declarations should appear before any inventory load, validation, NAS Reconcile, VM generation, or Service generation phase that requires the Dataset to exist.

Generated Dataset declarations should preserve deterministic names and paths so NAS resource ownership and cleanup remain predictable. The selected NAS Endpoint should come from the workflow invocation or resolved acceptance intent, so a run against a specific NAS Endpoint does not depend on a committed lab-specific Dataset YAML file.

By default, cleanup should remove generated Dataset YAML files after live acceptance resources have been destroyed. With `keep_on_fail=true`, generated Dataset YAML files should remain alongside the other generated artifacts so the operator can inspect or manually recover the run. The workflow must refuse to overwrite a pre-existing Dataset YAML file at the generated path unless that file is proven to be owned by the current generated acceptance run according to an existing accepted generated-artifact convention.

**Key interfaces:**
- Acceptance Policy shape — remains the durable declaration of the workflow's Dataset identity and reusable acceptance reservations.
- Ephemeral Dataset inventory shape — generated YAML should declare the workflow's Dataset name, selected NAS Endpoint, deterministic acceptance path, and `lifecycle: ephemeral`.
- Acceptance Test orchestration — should create generated Dataset YAML before inventory-dependent phases and include it in default cleanup and `keep_on_fail=true` preservation behavior.
- Inventory validation — ordinary fleet validation should not observe committed acceptance-only Dataset declarations when Acceptance Tests are not running.
- Generated artifact safety — Dataset YAML overwrite checks should match the conservative behavior used for generated VM, Service, Sibling SOPS, and Quadlet Fragment artifacts.

**Acceptance criteria:**
- [ ] NFS shared-mount acceptance generates its Ephemeral Dataset YAML before the workflow loads or validates inventory that references that Dataset.
- [ ] Service-layer acceptance generates its Ephemeral Dataset YAML before the workflow loads or validates inventory that references that Dataset.
- [ ] Generated Dataset YAML contains deterministic `name`, selected NAS Endpoint, deterministic path, and `lifecycle: ephemeral`.
- [ ] Acceptance Policies continue to durably name the Dataset identity used by each workflow.
- [ ] The workflows refuse to overwrite pre-existing Dataset YAML files at generated Dataset paths.
- [ ] Default cleanup removes generated Dataset YAML files after destroying generated live NAS and VM resources.
- [ ] `keep_on_fail=true` preserves generated Dataset YAML files for inspection.
- [ ] Committed ordinary Dataset inventory no longer contains the NFS shared-mount or Service-layer acceptance Ephemeral Dataset declarations.
- [ ] Automated tests cover generated Dataset YAML contents, phase ordering, overwrite refusal, cleanup, `keep_on_fail=true`, and absence of durable acceptance Dataset inventory.

**Out of scope:**
- Changing the Ephemeral Dataset create/destroy safety model.
- Changing ordinary adopted Dataset validation or allowing ordinary workflows to create or destroy Datasets.
- Replacing Acceptance Policies with generated files.
- Changing generated VM names, VMIDs, Service artifact names, or existing acceptance network/storage reservations except where needed to reference the generated Dataset declarations.
